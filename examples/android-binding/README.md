# Android Binding — Gradle + CMake integration example

A minimal Android-only project showing the complete JNI generation pipeline:

```
include/image_classifier.h          ← your C API
    │  --kotlin-from-header (Phase 1)
    ▼
src/ImageClassifier.kt              ← scaffolded Kotlin external fun stubs
    │  forward pass (Phase 2)
    ▼
generated/ImageClassifier_jni.gen.cpp  ← JNI C++ bindings, ready for CMake
```

## Inputs

### `include/image_classifier.h` — C API (Phase 1 input)

The native library's public C header.  Used by `--kotlin-from-header` to
scaffold the Kotlin `object ImageClassifier` with `external fun` declarations.
Edit this header when the native API changes, then re-run Phase 1.

### `src/ImageClassifier.kt` — Kotlin stubs (Phase 2 input)

Auto-generated by Phase 1.  You can also edit it by hand after generation
(e.g., rename a parameter) — Phase 2 reads this file directly.

| Function | Demonstrates |
|---|---|
| `imageClassifierCreate(String, Int, Boolean): Long` | string + primitives, returns opaque handle |
| `imageClassifierDestroy(Long)` | handle release, void return |
| `imageClassifierClassify(Long, ByteArray, Int, Int, FloatArray, Int): Int` | byte-array in, float-array out |
| `imageClassifierNumClasses(Long): Int` | simple scalar query |
| `imageClassifierGetLabel(Long, Int, String, Int): Int` | label lookup |
| `imageClassifierVersion(Long): String` | string return |
| `imageClassifierSetOption(Long, String, Float)` | key/value setter, void return |
| `imageClassifierGetOption(Long, String): Float` | key/value getter |

## Output

[`generated/ImageClassifier_jni.gen.cpp`](generated/ImageClassifier_jni.gen.cpp) —
JNI entry points with `Java_*` mangled names, argument marshalling, null/empty
guards, and TODO bodies for the native implementation.

## Gradle tasks

[`build.gradle.kts`](build.gradle.kts) exposes three tasks in the `jni` group:

| Task | What it does |
|---|---|
| `generateKotlinFromHeader` | Phase 1: `image_classifier.h` → `ImageClassifier.kt` |
| `generateJniBindings` | Phase 2: `ImageClassifier.kt` → `ImageClassifier_jni.gen.cpp` |
| `generateAll` | Both phases in order |

`generateJniBindings` is wired to run automatically before `externalNativeBuildDebug`
/ `externalNativeBuildRelease`.  The generator only writes files when content changes,
so no-op runs do not invalidate the CMake cache.

[`CMakeLists.txt`](CMakeLists.txt) shows how to add the generated `.cpp` to your
`add_library` target and point the include path at `jni-utils.h`.

## Phase 1 — scaffold Kotlin stubs from the C header

```bash
python3 scripts/jni-binding-generator.py \
    --kotlin-from-header examples/android-binding/include/image_classifier.h \
    --output examples/android-binding/src \
    --kotlin-package com.example.android.classifier
```

Or via Gradle:

```bash
./gradlew generateKotlinFromHeader
```

## Phase 2 — generate C++ JNI bindings from Kotlin stubs

```bash
python3 scripts/jni-binding-generator.py \
    --kotlin-source examples/android-binding/src/ImageClassifier.kt \
    --output examples/android-binding/generated
```

Or via Gradle:

```bash
./gradlew generateJniBindings
```

## Full pipeline in one shot

```bash
./gradlew generateAll
# or with the CLI:
python3 scripts/jni-binding-generator.py \
    --kotlin-from-header examples/android-binding/include/image_classifier.h \
    --output examples/android-binding/src \
    --kotlin-package com.example.android.classifier
python3 scripts/jni-binding-generator.py \
    --kotlin-source examples/android-binding/src/ImageClassifier.kt \
    --output examples/android-binding/generated
```

## Verify no drift

```bash
python3 scripts/jni-binding-generator.py \

    --output examples/android-binding/generated \
    --check
```
