# iOS App

The iOS target uses Kotlin/Native — no JNI is involved.

## Generated cinterop files

[`src/nativeInterop/cinterop/NativeBridgeJni.def`](src/nativeInterop/cinterop/NativeBridgeJni.def)
and [`src/nativeInterop/cinterop/include/NativeBridgeJni.h`](src/nativeInterop/cinterop/include/NativeBridgeJni.h)
were generated by `--ios-cinterop`:

```bash
python3 scripts/jni-binding-generator.py \
    --kotlin-source examples/kmp-binding/shared/src/androidMain/kotlin \
    --output examples/kmp-binding/androidApp/src/main/cpp/generated \
    --ios-cinterop examples/kmp-binding/iosApp/src/nativeInterop/cinterop
```

## After generation

1. Edit `include/NativeBridgeJni.h` — replace the placeholder function
   signatures with the real C API exported by your native library.
2. Fill in `staticLibraries` / `libraryPaths` in `NativeBridgeJni.def`.
3. Wire the cinterop in `shared/build.gradle.kts` under `iosMain`:
   ```kotlin
   iosMain {
       cinterops {
           val native_bridge_jni by creating {
               defFile("src/nativeInterop/cinterop/NativeBridgeJni.def")
           }
       }
   }
   ```
4. Replace stub bodies in
   `shared/src/iosMain/kotlin/com/example/kmmbinding/NativeBridge.ios.kt`
   with real calls to the cinterop bindings.
5. Open `iosApp/iosApp.xcodeproj` in Xcode (generated by the KMP Gradle plugin
   after `./gradlew :shared:linkDebugFrameworkIosSimulatorArm64`).

The JNI binding generator does **not** process `iosMain` sources — it only
covers `androidMain` and `desktopMain`.
